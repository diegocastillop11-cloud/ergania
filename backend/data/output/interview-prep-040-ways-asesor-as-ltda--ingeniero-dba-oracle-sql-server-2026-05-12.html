## 2. PREGUNTAS TÉCNICAS DEL ROL

### 2.1 "¿Cómo identificas y resuelves problemas de performance en SQL Server?"

**Cómo abordarla:**

"Mi enfoque es sistemático:

1. **Identificación:**
   - Uso `sp_WhoIsActive` o `sp_who2` para ver sesiones activas y bloqueadas
   - Reviso el Activity Monitor y los wait stats con `sys.dm_os_wait_stats`
   - Analizo el plan de ejecución de queries lentas con `SET STATISTICS IO ON` y `SET STATISTICS TIME ON`

2. **Diagnóstico:**
   - Identifico queries que hacen table scans en lugar de index seeks
   - Reviso si hay índices faltantes con `sys.dm_db_missing_index_details`
   - Verifico estadísticas desactualizadas

3. **Resolución:**
   - Creo índices apropiados (clusterizados, no clusterizados, filtered)
   - Reescribo queries: uso EXISTS en lugar de IN cuando es más eficiente, evito funciones en WHERE sobre columnas indexadas
   - Optimizo stored procedures: uso table variables o temp tables según el volumen
   - Actualizo estadísticas con `UPDATE STATISTICS`

4. **Validación:**
   - Mido tiempos antes y después
   - Monitoreo en producción para asegurar mejora sostenida

Un caso real: reduje un reporte de 45 segundos a 3 segundos simplemente agregando un índice compuesto y reescribiendo un JOIN innecesario."

### 2.2 "Explica la diferencia entre clustered y non-clustered index y cuándo usar cada uno"

**Cómo abordarla:**

"**Clustered Index:**
- Reordena físicamente las filas en disco según la clave del índice
- Solo puede haber uno por tabla
- Por defecto, SQL Server lo crea en la Primary Key
- Ideal para columnas que se consultan en rangos (fechas, IDs secuenciales)

**Non-Clustered Index:**
- Crea una estructura separada con punteros a las filas reales
- Puedes tener hasta 999 por tabla (aunque no es recomendable tantos)
- Ideal para columnas que se filtran frecuentemente pero no definen el orden natural de la tabla

**Cuándo uso cada uno:**
- **Clustered:** En la columna que define el orden lógico de la tabla (ej: FechaCreacion en una tabla de transacciones)
- **Non-Clustered:** En columnas de filtro frecuente (ej: Email en tabla de Usuarios, Estado en tabla de Pedidos)

**Tip adicional:** Si una consulta cubre todas las columnas que necesita con un non-clustered index (covering index), SQL Server no necesita ir a la tabla, mejorando aún más el performance."

### 2.3 "¿Cómo realizas backups y cuál es tu estrategia de recuperación?"

**Cómo abordarla:**

"Mi estrategia depende del RPO (Recovery Point Objective) y RTO (Recovery Time Objective) del negocio, pero típicamente implemento:

**Modelo de Backup:**
- **Full Backup:** Domingos a las 02:00 AM (baseline completo)
- **Differential Backup:** Miércoles a las 02:00 AM (solo cambios desde el último full)
- **Transaction Log Backup:** Cada 15-30 minutos en horario laboral (mínima pérdida de datos)

**Estrategia de Recuperación:**
1. Restauro el último Full Backup
2. Aplico el último Differential si existe
3. Aplico todos los Transaction Logs posteriores en secuencia con `WITH NORECOVERY`
4. Ejecuto el último log con `WITH RECOVERY`

**Validación:**
- Pruebo restauraciones mensualmente en un ambiente de QA
- Verifico integridad con `DBCC CHECKDB` después de cada full backup
- Documento los procedimientos de recuperación paso a paso

En Punto Ticket, esta estrategia nos permitía recuperar datos con pérdida máxima de 15 minutos, lo cual estaba dentro del acuerdo de nivel de servicio."

### 2.4 "¿Qué diferencias clave hay entre Oracle y SQL Server en términos de administración?"

**Cómo abordarla:**

"He trabajado con ambos y algunas diferencias importantes son:

**Arquitectura:**
- **Oracle:** Usa instancias y esquemas (schemas) dentro de bases de datos. Un schema es como un contenedor de objetos de un usuario.
- **SQL Server:** Una instancia contiene múltiples bases de datos independientes.

**Gestión de Memoria:**
- **Oracle:** SGA (System Global Area) y PGA (Program Global Area) - configuración más manual.
- **SQL Server:** Buffer Pool y Plan Cache - más automático, aunque se puede ajustar.

**Backup:**
- **Oracle:** RMAN es la herramienta estándar, con conceptos como archivelog mode.
- **SQL Server:** Backups nativos con T-SQL o mediante GUI, más intuitivo inicialmente.

**Lenguaje:**
- **Oracle:** PL/SQL
- **SQL Server:** T-SQL

**Privilegios:**
- **Oracle:** Roles y privilegios más granulares y complejos.
- **SQL Server:** Roles de base de datos y server roles más simples.

Si bien mi expertise es SQL Server, entiendo la lógica de Oracle y me adapto rápido. De hecho, cuando trabajé en Imperial, logré resolver incidencias complejas en Oracle consultando documentación y aplicando mis fundamentos de bases de datos relacionales."

### 2.5 "¿Cómo gestionas la seguridad en bases de datos?"

**Cómo abordarla:**

"La seguridad en BD es multicapa:

**1. Autenticación:**
- Prefiero Windows Authentication sobre SQL Authentication (más segura, integrada con AD)
- Si uso SQL Auth, fuerzo políticas de contraseña fuertes con `CHECK_POLICY = ON`

**2. Autorización:**
- Principio de menor privilegio: los usuarios solo tienen permisos sobre lo que necesitan
- Uso roles de base de datos (`db_datareader`, `db_datawriter`) en lugar de permisos directos
- Creo roles personalizados para grupos de usuarios con necesidades similares

**3. Encriptación:**
- Transparent Data Encryption (TDE) para cifrar datos en reposo
- Always Encrypted para datos sensibles (ej: números de tarjetas)
- Certificados SSL/TLS para conexiones

**4. Auditoría:**
- Habilito SQL Server Audit para rastrear accesos y cambios críticos
- Monitoreo logins fallidos y actividad sospechosa

**5. Mantenimiento:**
- Reviso y elimino cuentas inactivas regularmente
- Actualizo parches de seguridad de SQL Server
- Restrinjo acceso directo a producción: solo vía jump servers o VPNs corporativas

En Punto Ticket, implementé auditoría sobre tablas de ventas para cumplir con requisitos de trazabilidad financiera."

"Analista de Base de Datos Senior" es genérico. No especifica SQL Server ni Oracle en el título. Si busco "Ingeniero DBA Oracle" en LinkedIn o en un ATS, este CV puede no aparecer.

**Solución:** Cambia el título a: **"Ingeniero DBA | SQL Server · Oracle · Azure | Alta Concurrencia & Automatización"**

---

**2. Falta de keywords críticas para ATS:**
No aparecen términos como:
- Performance tuning
- High availability (HA)
- Disaster recovery
- Índices compuestos
- Transact-SQL
- PL/SQL
- RMAN
- Always On
- Replicación

Si un reclutador filtra por "RMAN" o "Always On" en su ATS, este CV no va a matchear.

**Solución:** Agrega una sección **"Competencias Técnicas"** antes de Experiencia Profesional con estas keywords, incluso si tu experiencia es básica. Ejemplo:

```
## Competencias Técnicas Clave
- Administración SQL Server 2012-2019 (instancias productivas, performance tuning, índices)
- Fundamentos de Oracle Database (PL/SQL, RMAN, tablespaces, roles)
- Alta disponibilidad y recuperación (backups full/differential/log, disaster recovery planning)
- Automatización (T-SQL scripts, SQL Agent Jobs, PowerShell)
- Integración de datos (HTML, JSON, XML)
- Cloud (Microsoft Azure SQL, migraciones on-premise to cloud)
```
